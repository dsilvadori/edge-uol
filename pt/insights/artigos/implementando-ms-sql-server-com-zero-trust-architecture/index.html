Implementando MS SQL Server com Zero Trust Architecture

Tecnologia

Autonomous Operations

Postado em: 17/05/2023
Tempo de leitura: 6 minutos
Por: Cauã Prado Pastore

A cada dia nos tornamos mais dependentes de soluções digitais e da internet, e com isso ficamos expostos às ameaças virtuais como vírus, ransomware, trojan (Cavalo de Tróia), etc.

Com esse cenário em mente, todo profissional de tecnologia da informação deve ter boas noções de segurança. Aqui na Edge UOL não é diferente. Por administrarmos ambientes complexos e extremamente críticos em vários segmentos, sendo muitos deles instituições financeiras, levamos bem a sério todas as melhores práticas de segurança da informação.

Nesse artigo explicaremos como um DBA pode implementar uma instância do MS SQL Server adotando o conceito de Zero Trust Architecture (ZTA) para garantir um ambiente seguro que, caso seja comprometido, isole o ataque antes de ser elevado ao ponto de chegar a exfiltração, alteração ou destruição de dados.

A Arquitetura Zero Trust é um modelo de segurança da informação que assume que nenhum acesso deve ser concedido por padrão e sim somente quando necessário e respeitando as especificações de níveis de acesso. A ênfase está na verificação contínua e na aplicação de políticas de acesso granulares para garantir a segurança dos dados e sistemas.

Implementar o SQL Server em um ambiente Zero Trust requer a adoção de várias práticas recomendadas e parametrização de segurança para garantir a proteção adequada das instâncias e dos dados armazenados. Abaixo discutimos algumas delas.

Planejamento e Implementação

  • Segmentação de Rede: Quando possível separe as instâncias do SQL Server e servidores de aplicação em subnets (sub-redes) diferentes aplicando regras de firewall para controlar o fluxo de dados entre os segmentos somente através das portas dos serviços de banco. Isso limita o campo de um ataque e inibe a propagação de ameaças virtuais.
  • Princípio do Menor Privilégio: É muito comum vermos o time de aplicação pedir para que seja mapeada a role db_owner para o usuário da aplicação. Questione e tire proveito do princípio do menor privilégio garantindo que os usuários nominais e os usuários de aplicação tenham apenas as permissões necessárias para executar suas tarefas. Isso reduz o risco de exploração de logins comprometidos ou acesso não autorizado.
  • Autenticação e Autorização: Use a autenticação multifator (MFA) e Windows Authentication para validar a identidade dos usuários antes de conceder acesso às instâncias do SQL Server. Para saber mais, clique aqui.

Configuração e Proteção de Dados

  • Criptografia de Dados em armazenamento e em movimento: Proteja os dados armazenados usando Transparent Data Encryption (TDE) e criptografe os dados em movimento usando SSL/TLS. Assim garantimos a confidencialidade e integridade dos dados armazenados e transmitidos. Veja mais em aqui.
  • Gerenciamento de Chaves: Implemente uma solução de gerenciamento de chaves centralizada e segura, como o Azure Key Vault ou uma solução de HSM, para armazenar e gerenciar as chaves de criptografia usadas para proteger os dados do SQL Server.
  • Monitoramento e Auditoria: Crie um plano de monitoramento e auditoria para logar o acesso e as atividades dos usuários nas instâncias do SQL Server. Utilize logs e alguma ferramenta SIEM para correlacionar eventos e identificar atividades suspeitas.

Detecção e Resposta a Ameaças

  • Análise Comportamental: Incorpore soluções de análise comportamental para monitorar o comportamento dos usuários e identificar anomalias, como tentativas de acesso fora do horário normal de trabalho, atividades incomuns de consulta ou tentativas de exfiltração de dados.
  • Resposta a Incidentes: Desenvolva um plano de resposta a incidentes para lidar com violações de segurança, incluindo a identificação, contenção, erradicação, recuperação e análise pós-incidente. Integre a resposta a incidentes com as ferramentas de monitoramento e detecção para garantir uma resposta rápida e eficaz às ameaças.

Continuidade dos Negócios e Recuperação de Desastres

  • Backup e Restore: Implemente uma estratégia de backup e restore (restauração) com dados off site e imutáveis para garantir a recuperação rápida e completa dos dados e instâncias do SQL Server em caso de falha de hardware, corrupção de dados ou ataque de ransomware.
  • Teste de Restore: Realize testes periódicos de restore para verificar a eficácia das soluções de backup e restore e garantir que os processos de recuperação sejam eficientes e alinhados com os objetivos de recuperação de tempo (RTO) e ponto (RPO) da organização.

Manutenção e Aplicação de Patch

  • Patch Management: Mantenha as instâncias do SQL Server atualizadas com os patches de segurança mais recentes e atualizações de software para corrigir vulnerabilidades conhecidas e reduzir a superfície de ataque.
  • Avaliações de Segurança Regulares: Conduza avaliações de segurança regulares para identificar e remediar possíveis vulnerabilidades nas instâncias do SQL Server, infraestrutura de rede e políticas de acesso.
    Ao adotar as práticas recomendadas nesse artigo, como segmentação de rede, elevar os níveis de autenticação, criptografia de dados e monitoramento contínuo, os DBAs e profissionais de segurança da informação criam um ambiente mais seguro e resiliente para o gerenciamento dos dados corporativos. Mas não se esqueça que além dos pontos de ZTA existe toda a parametrização costumeira de SO e SQL que também têm vários pontos de segurança da informação e não podem ser deixadas de lado.
